# FIP Positioning

## What this document is for

This document keeps FIP aligned across implementation, patent direction, and grant framing.

It is meant to help code and architecture work stay consistent with:

- the earlier provisional invention basis for FIP
- the current standalone product direction
- the intended standalone non-provisional filing path
- the NIST-targeted funding and positioning lane

## Status note

The earlier provisional material is the historical invention basis for FIP.

The standalone non-provisional planned for this year is expected to be based on the current FIP build, not treated as a verbatim copy of the earlier provisional language.

That means:

- preserve invention continuity
- preserve the core principles
- do not assume every provisional term is final
- let the current build drive the standalone filing shape

## Original invention basis

The earlier provisional basis establishes FIP around these core principles:

- fractured identity storage
- secure sharded identity representation
- controlled reconstruction
- reconstruction only through authorized runtime conditions
- volatile/session-lifetime identity assembly
- anti-reconstructability by design
- write-back with re-sharding
- privacy/security through architecture, not just access control

These principles remain central.

## What FIP is

FIP is a governed identity reconstruction layer for AI systems.

It enables bounded, auditable reconstruction of fractured identity material for active runtime use without requiring uncontrolled full-context exposure.

## What FIP is not

FIP is not:

- generic chatbot memory
- a profile blob
- vector recall with branding
- a social-memory feature
- a monolithic personalization database
- a provider-owned continuity system

## Product framing

Preferred product framing:

- governed identity reconstruction
- bounded context assembly
- fractured identity infrastructure
- auditable reconstruction control
- identity exposure minimization
- trustworthy AI identity layer

## Grant framing

FIP is being developed toward a NIST-aligned positioning.

That means FIP should be framed in terms of:

- trustworthy AI
- bounded exposure
- controlled reconstruction
- provenance
- auditability
- integrity validation
- privacy/security by architecture
- measurement-friendly control surfaces
- infrastructure for governed personalization or continuity

## Why NIST fits

NIST is a better fit for FIP than broad assistant language because FIP is not mainly about convenience. It is about controlled, explainable, bounded, and defensible identity handling for AI systems.

FIP should naturally align with language such as:

- trustworthy AI infrastructure
- bounded identity-context exposure
- auditable inclusion and exclusion decisions
- governed reconstruction
- anti-reconstructability design
- integrity-aware identity control

## Code alignment implications

Implementation work should reinforce the invention and funding lane.

This means code should:

- preserve fractured identity concepts
- avoid monolithic assembled-at-rest assumptions
- keep reconstruction bounded and explicit
- preserve protected-type default denial
- preserve provenance for reconstruction decisions
- avoid provider coupling in core
- use architecture that supports future audit and integrity claims

## Patent-alignment implications

Code should not unnecessarily narrow the future standalone filing.

That means:

- keep abstractions clean
- do not hardcode accidental implementation assumptions as if they are invention boundaries
- preserve the core fractured-storage and controlled-reconstruction principles
- let the current build mature into the standalone filing basis

## Practical instruction for contributors

When making architecture-sensitive changes:

1. preserve the fractured identity model
2. preserve bounded reconstruction behavior
3. preserve protected-class controls
4. preserve provenance
5. avoid drifting into generic memory design
6. keep FIP understandable as a standalone product and filing lane
